Check content and loop with AI

Runs anywhere

No project needed — kapi check runs ad-hoc on any source file.

Goal: verify content before it ships — flag over-long strings, forbidden phrases, brand drift, and structural faults — and fix problems in a tightly scoped loop: either interactively from the command line or automatically by an AI assistant working over MCP.

kapi check is the entry point. It reads a file, runs a content-first checkset, and returns one machine-readable kapi.check/v1 Report. It never modifies content; editing the flagged blocks is kapi apply's job.

For the concepts, see Checks and Edit content; this recipe walks the task.

Run a check​

kapi check src/locales/en.json

kapi extracts the text from the file, runs the default checkset — hygiene, length, patterns, and brand vocabulary — and writes the result to stdout. With --json, it emits the structured Report instead:

kapi check src/locales/en.json --json
{
  "schema": "kapi.check/v1",
  "pass": false,
  "summary": {
    "score": 74,
    "critical": 0,
    "major": 2,
    "minor": 3
  },
  "gate": "major",
  "findings": [
    {
      "rule": "length.max-chars-exceeded",
      "severity": "major",
      "message": "Block exceeds the 120-character limit (148 chars).",
      "suggestion": "Shorten to 120 characters.",
      "location": { "file": "src/locales/en.json", "block": "onboarding.welcome" }
    },
    {
      "rule": "hygiene.doubled-word",
      "severity": "major",
      "message": "Doubled word: \"the the\".",
      "location": { "file": "src/locales/en.json", "block": "settings.save_prompt" }
    }
  ]
}

(See Checks for the full kapi.check/v1 Report schema — every field, the score, the gate, and the finding model.)

Add length, pattern, and brand rules​

Pass constraints inline, or collect them in a named pack:

# Length budget + forbidden phrase
kapi check src/locales/en.json \
  --max-chars 120 \
  --max-words 30 \
  --forbid "click here"

# A named pack that bundles all house-style rules
kapi check src/locales/en.json --pack house-style

# Or point at a profile file
kapi check src/locales/en.json --profile-file .kapi/check-profile.yaml

--require RE asserts the pattern must appear (useful for mandatory disclaimers or structured string formats). --forbid RE fails any block containing the pattern.

Gate in CI​

kapi check exits 3 when the gate trips — a severity at or above the gate threshold appears in the findings. This integrates directly with CI:

.github/workflows/content.yml
- name: Check content
  run: kapi check src/locales/en.json --pack house-style --json

The exit codes:

ExitMeaning
0All findings are below the gate threshold — pass
3At least one finding reached the gate — gate fail
1Operational error (file not found, format unrecognised)

--strict tightens the gate to minor severity (any finding fails). --lenient raises it to critical only. --no-fail always exits 0, turning the check into a report-only run — useful inside an assistant fix-loop where the orchestrator reads the Report and decides whether to iterate.

Fold in structure and encoding findings​

By default, kapi check focuses on content rules. To also surface format-reader findings — malformed XML, invalid UTF-8, unrecognised character sets, rejected JSON — pass --validate:

# Add structure/encoding findings to the Report (does not gate)
kapi check src/locales/en.json --validate report

# Gate on structure/encoding findings too
kapi check src/locales/en.json --validate strict

With --validate report, structure.* and encoding.* findings appear in the Report but do not affect the gate. With --validate strict, they gate like any other finding at their severity level.

Analyse before you check​

Two commands give a quick read of a file before running a full check.

kapi inspect returns one record per content block, useful for RAG pipelines or feeding a snapshot to an AI:

kapi inspect src/locales/en.json
[
  {
    "file": "src/locales/en.json",
    "number": 1,
    "id": "onboarding.welcome",
    "content_hash": "a3f2...",
    "role": "block",
    "level": 0,
    "text": "Welcome to the app!"
  }
]

Add --jsonl for one JSON object per line (friendlier for streaming). Specify -f fmt to override format detection.

kapi stats gives a file-level overview before you commit to a full check pass:

kapi stats src/locales/en.json
kapi stats src/locales/en.json --json
{
  "total": {
    "blocks": 42,
    "translatable": 38,
    "non_translatable": 4,
    "words": 312,
    "characters": 2048,
    "characters_no_space": 1740,
    "segments": 54,
    "by_role": { "block": 38, "data": 4 }
  }
}

The typical analysis sequence is: kapi inspect to browse the blocks, kapi stats to understand the volume, then kapi check to run the gates.

Opt in to bilingual checks​

Content-side checks run without a target file. To also check placeholder integrity, do-not-translate compliance, and terminology enforcement against a translated target, add --target and --target-lang:

kapi check src/locales/en.json \
  --target src/locales/fr.json \
  --target-lang fr

This folds bilingual findings into the same Report — one pass, one exit code, one kapi.check/v1 schema.

The author → check → revise loop​

When a finding points at a block, read the blocks, rewrite the flagged ones yourself, and write them back with kapi apply. The file format is preserved; only the text changes, and no model is involved:

# Read the blocks, edit the flagged ones, preview the change
kapi inspect src/locales/en.json --jsonl > blocks.jsonl
# rewrite each changed block's "text" (keep the <x id="…"/> tags), saving them as
# content entries to edits.jsonl — you (or your assistant) do this, no command
kapi apply edits.jsonl --diff

# Apply in place
kapi apply edits.jsonl

# Re-check
kapi check src/locales/en.json --max-chars 120

Repeat until the exit is 0. This is the same loop an AI assistant runs over MCP.

Drive the loop from an AI assistant​

When kapi is configured as an MCP server (kapi mcp), an assistant has four relevant tools:

MCP toolWhat it does
extract_contentRead a file's blocks (id, content_hash, placeholder text) — the read leg
check_fileRun kapi check on a file; returns the kapi.check/v1 Report
check_textCheck a text string inline without a file
apply_editsApply a typed change-set of content edits in place — the write leg, no provider
brand_checkRun brand vocabulary and voice rules against a file

A typical assistant turn:

  1. Author or edit content in a file.
  2. Call check_file — receives the Report with stable rule ids and block locations.
  3. For each failing block, rewrite its text and send it back with apply_edits (a content entry carrying the block's content_hash).
  4. Call check_file again. Iterate until pass is true.

Because rule ids are stable across runs, the assistant can track which findings it resolved and which it introduced, and stop when the Report is clean. The loop requires no human intervention and works identically over files of any format kapi reads.

See MCP server for setup and the full tool list.

In a project​

In a .kapi project, kapi check runs against the recipe's declared content and locales — kapi check with no file argument checks the project's source, and a check can run as a step in a flow you launch with kapi run. The check profile, length budgets, and brand vocabulary come from the recipe instead of inline flags, so the same gate applies everywhere it runs. See Create your first project.
