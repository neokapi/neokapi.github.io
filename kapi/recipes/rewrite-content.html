Rewrite content with AI, faithfully

Requires an AI provider

kapi rewrite calls an LLM. Configure a provider first — kapi credentials add — or pass --provider on the command line to pick one explicitly.

Goal: edit the text inside a file without touching its structure. The source file may be a Word document, a JSON catalog, a Markdown guide, an HTML page, or any other format kapi reads. The instruction is plain language. The output is a new copy of the same file — same keys, same styles, same inline codes — with only the prose changed according to what you asked.

Common uses: simplify technical documentation, fix off-brand copy, standardize terminology across a catalog, tighten prose to fit a length budget.

Assess the file first​

Before rewriting, it helps to see how much text you are dealing with. kapi stats reads the file and reports block counts, word totals, and character totals without modifying anything:

kapi stats guide.md
# Machine-readable output for scripting
kapi stats guide.md --json

To read the text block by block — for feeding to another tool or for your own inspection — use kapi inspect. It returns one record per extracted block, including the block's role, nesting level, and a content hash:

kapi inspect guide.md
kapi inspect guide.md --jsonl        # one JSON object per line, for streaming
kapi inspect guide.md -f markdown    # override format detection

Neither command modifies the file.

Rewrite from a plain-language instruction​

Pass the instruction and the file. kapi reads the file, extracts the text into blocks, sends them to the LLM with the instruction, and writes the results back:

kapi rewrite --instruction "make it more concise" guide.md

The output lands next to the input by default (suffixed .out or as a new file, depending on how the format writer resolves the path). Pass a provider explicitly if you have more than one configured:

kapi rewrite --instruction "standardize 'log in' to 'sign in' throughout" \
  --provider anthropic \
  content/docs/auth.md

Multiple files work in one call:

kapi rewrite --instruction "simplify to plain language, reading level 8" \
  docs/*.md

Review before applying​

Add --diff to perform a dry run. kapi runs the rewrite but prints a unified diff instead of writing the output, so you can read every proposed change before committing to it:

kapi rewrite --instruction "make it more concise" --diff guide.md

The diff shows only the text that changed. Structure — headings, list markers, code fences, inline codes — appears unchanged because kapi does not touch it.

Apply in place​

Once the diff looks right, rewrite the file in place with -i:

kapi rewrite --instruction "make it more concise" -i guide.md

To keep the original as a backup, append an extension to --in-place:

kapi rewrite --instruction "make it more concise" --in-place=.bak guide.md

This writes guide.md.bak (the original) and updates guide.md in place.

What "faithfully" means​

The rewrite is faithful in a precise sense: kapi's format reader extracts the text blocks from the file and the writer puts them back into the exact same structural slots. Everything outside the extracted text is reproduced byte for byte.

For a Word document (.docx), this means character styles, table borders, tracked changes, and image positions are not affected — only the paragraph text changes. For a JSON catalog, the keys, nesting, and non-string values are untouched — only string values change. For a Markdown file, headings, fences, and inline codes are preserved.

This is the same round-trip guarantee that every kapi format writer provides. See Tools for the general model, and the format reference for per-format specifics.

Verify the result​

After rewriting, run kapi check to confirm the result meets your content rules. The check is deterministic: it reads the file and reports findings without modifying it:

kapi check guide.md

Apply rules as flags when you do not have a check profile:

# Flag strings over 500 characters or containing a forbidden term
kapi check guide.md --max-chars 500 --forbid "utilize"

Use --profile to run a named checkset, or --profile-file for a local YAML:

kapi check guide.md --profile company-voice

The command exits 0 when all checks pass and 3 when the gate fails. The structured report is available via --json:

kapi check guide.md --json

The report schema is kapi.check/v1 and includes a score, severity-bucketed findings, and per-block locations. See Checks for the full check model.

In an AI assistant's fix-loop​

When kapi is wired as an MCP server (kapi mcp), an AI assistant can drive the same workflow through structured tool calls. The relevant tools are:

MCP toolWhat it does
check_fileRun the checkset against a file; returns a kapi.check/v1 Report with per-block finding locations
check_textRun the checkset against a text snippet
rewrite_fileRewrite the text inside a file from an instruction (and optional brand profile), structure preserved
brand_checkScore a text against a brand voice profile

A typical agent pass: the assistant calls check_file to read the findings, then calls rewrite_file with an instruction derived from those findings, then calls check_file again to confirm the gate passes. The loop is self-correcting: each check pinpoints the exact block and rule, so the rewrite instruction can be specific.

rewrite_file uses the same faithful round-trip as kapi rewrite: only text changes, structure is preserved. See MCP server for setup and the full tool list.

For the agent-skills path (an AI calling the kapi CLI directly rather than the MCP server), kapi inspect with --jsonl is a convenient way to feed the extracted text to the model block by block before issuing a targeted rewrite.
