Bowrain

Bowrain is the server-side platform companion to kapi. Kapi gives a solo builder or a small team everything they need locally — extracting content, enforcing brand voice and terminology, leveraging translation memory, and localizing into every language and format, all from files you own. Bowrain is where a team hosts, governs, and stewards that work over time: shared profiles and memory, collaborative editing, connectors to the systems content already lives in, automation, and version history.

It is to kapi what GitHub is to git. The toolchain stays open and runs anywhere; Bowrain adds the persistent, multi-user, governed layer an organization needs.

A team shares one workspace through a browser and a native desktop app — editing, terminology, translation memory, presence, and reviews all live on the server.

What Bowrain adds​

Connect your content — from a CMS, a design tool, a git repository, or a kapi project — and the platform adds what a single checkout or a single source system cannot:

Shared, versioned governance​

One brand-voice profile, terminology base, and translation memory, held on the server and shared across everyone — writers, translators, and AI tools — instead of living in each person's checkout. Changes are versioned and auditable: who approved a term, when a profile changed, what a translation used to say. And the profile learns: repeated corrections become versioned, enforced checks, so the rules a team discovers by correcting real output are captured instead of lost — see Brand voice & corrections.

Collaboration​

A web editor and a desktop app connect to the same server. Translators work in Visual and Table views with translation memory and terminology context; edits, presence, and live updates propagate to every client without a refresh — see Real-time collaboration.

Connectors​

Bowrain syncs against the systems content already lives in — a CMS (WordPress, HubSpot), a design tool (Figma), a code repository (GitHub), or a developer's local files via kapi. Most of these pull content in server-side, with no local checkout: source flows into the governed workspace, gets translated and checked, and publishes back to where it came from. kapi is the local-files/git connector — one category among several, not the whole story. See Connectors.

AI that stays on-brand​

The workspace's governed terminology, translation memory, and brand-voice checks are the context an AI assistant reads — over MCP — so generated and translated copy is on-voice from the first draft, and corrections flow back into the shared, enforced checks.

Automation​

Server-side rules run flows when content arrives, a quality gate passes, or a connector syncs. Local rules in a .kapi recipe give a developer guardrails; server rules give a team shared, event-driven workflows.

Stewardship over time​

A persistent, versioned content store keeps the history of every block, branch, and translation, so the work compounds: terminology decisions, brand-voice corrections, and remembered translations stay with the project as people and tools come and go.

One command to sync​

With the plugin installed, a kapi project whose recipe declares a server: block can push and pull in a single step:

kapi sync

kapi sync pushes changed content to the server, waits for the triggered flows (translation, QA, terminology) to finish, and pulls the results back — one invocation from code change to translated files. In CI it replaces a multi-job pipeline.

Content does not have to start locally, though: a connector to a CMS, a design tool, or a git repository pulls source into the workspace server-side, with no checkout at all. kapi sync is the path for a developer working from a local codebase — one of several ways content reaches the platform.

How it fits together​

Content flows in from any connector — a CMS, a design tool, git, or a developer's local files via kapi. The server governs and automates, people and AI tools review, and results publish back.

Content enters the workspace from a connector — a CMS, a design tool, a git repository, or a developer's local files via kapi sync. Server automation orchestrates translation and checks; translators (and AI tools, over MCP) work in the web or desktop editor with translation memory and terminology support; and the content store keeps the history.

Parts of the platform​

  • Server — workspaces, authentication, the versioned content store, automation, and connectors to the systems content already lives in.
  • Connectors — sync content from a CMS, a design tool, a git repository, or local files. CMS and design connectors run server-side with no checkout; kapi is the local-files/git connector.
  • Web editor — a browser-based translation workbench with Visual and Table views, translation memory, and terminology.
  • Desktop app — a native cross-platform editor with offline support.
  • Project sync (kapi) — the bowrain plugin for kapi, run as kapi <command>, which syncs a local .kapi project with a server — one connector among several.

Just need the local tools?​

kapi runs on its own — extraction, brand and terminology checks, translation, pseudo-translation, QA — directly on files, with no server or account. Reach for Bowrain when a team needs the shared, governed, and durable layer on top. See the kapi documentation.
