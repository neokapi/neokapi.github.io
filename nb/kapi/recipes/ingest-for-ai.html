Feed content to your AI

Runs anywhere

No project needed — kapi stats, kapi inspect, kapi check, and kapi rewrite run ad-hoc on any source document.

Goal: turn any document — Word, PDF, JSON, Markdown, HTML — into a stream of clean, anchored content blocks an AI assistant or RAG pipeline can read without knowing anything about the source format. No translation is involved. Each block carries a stable content_hash anchor that lets you find and rewrite that block later, in the same file, without touching the surrounding markup.

For the concepts behind anchored blocks and content checks, see Content model and Checks; this recipe walks the task.

Step 1 — survey before you ingest​

Before feeding a document to a model, kapi stats gives you a format-agnostic size and structure overview: block counts, word and character counts, and a breakdown by structural role (heading, paragraph, list-item, table-cell, and so on). Use it to estimate token budget and decide how to chunk.

kapi stats report.docx
Blocks:                     84
  translatable:             76
  non-translatable:          8
Words:                   11342
Characters:              71890
  (no spaces):            60412
Segments:                  289

By role:
  paragraph             47
  heading                9
  list-item             20

For scripting — sizing content inside a loop or a pipeline — add --json:

kapi stats --json report.docx docs/*.md | jq '.total.words'

The total record in the JSON output has the same shape as the per-file entries: blocks, translatable, non_translatable, words, characters, characters_no_space, segments, and by_role. Pass -f to declare the input format when auto-detection is ambiguous.

Step 2 — extract blocks with anchors​

kapi inspect reads a file through its format-aware parser and emits one record per content block. Every record contains the plain text, a stable content_hash (SHA-256 over the normalized source text), the block's structural role, and its nesting level. The format collapses: a Word document, a JSON catalog, and a Markdown page all yield the same record shape.

kapi inspect report.docx

Produces a JSON array:

[
  {
    "file": "report.docx",
    "number": 1,
    "id": "b1",
    "content_hash": "e3b0c44298fc1c149afb4c8996fb92427ae41e4649b934ca495991b7852b855",
    "role": "heading",
    "level": 1,
    "text": "Introduction"
  },
  {
    "file": "report.docx",
    "number": 2,
    "id": "b2",
    "content_hash": "6b86b273ff34fce19d6b804eff5a3f5747ada4eaa22f1d49c01e52ddb7875b4d",
    "role": "paragraph",
    "level": 0,
    "text": "This document describes the migration path."
  }
]

For large documents or pipeline ingestion, --jsonl streams one object per line (newline-delimited JSON), which is easier to pipe incrementally than a closed array:

kapi inspect --jsonl report.docx docs/*.md | \
  jq -c '{hash: .content_hash, role: .role, text: .text}'

Pass -f to declare the input format explicitly:

cat page.html | kapi inspect -f html --jsonl

The anchor and write-back contract​

The content_hash is a stable retrieval key derived from the block's text, not its position: a RAG pipeline that indexes blocks by their content_hash can later look up the original block when an AI assistant produces an edit. (See Content model for how blocks and anchors are represented.)

The write-back path closes the loop:

  1. Read — kapi inspect --jsonl extracts blocks with anchors.
  2. Retrieve — the AI finds the relevant block by content_hash or id.
  3. Edit — the AI produces a revised version of the text.
  4. Write-back — kapi rewrite or the MCP rewrite_file tool applies the edit back to the file through the same format-aware round-trip, leaving structure and inline markup intact.

The write-back is a faithful, format-aware round-trip: only the text changes, and a block whose inline codes would be corrupted is left unchanged rather than written with unbalanced markup. See Rewrite tool for the round-trip contract.

Checking content before or after editing​

kapi check runs a set of content checks over any file and returns a structured kapi.check/v1 Report an AI assistant can act on without parsing free-form output.

kapi check report.docx

Add length or pattern constraints:

kapi check report.docx --max-chars 300 --forbid "TODO"

Bind a brand profile to also run vocabulary and brand-voice checks:

kapi check report.docx --pack marketing-blog
kapi check report.docx --profile my-profile
kapi check report.docx --profile-file ./brand.yaml

The --json flag emits the kapi.check/v1 Report:

kapi check --json report.docx | jq '.findings[] | select(.severity == "critical")'

The report shape:

{
  "schema": "kapi.check/v1",
  "pass": false,
  "summary": { "score": 74, "critical": 0, "major": 2, "minor": 3 },
  "gate": { "failed": ["major findings exceed limit"] },
  "findings": [
    {
      "rule": "length.max-chars-exceeded",
      "severity": "major",
      "message": "Block exceeds 300 characters (actual: 412).",
      "suggestion": "Split or shorten the block.",
      "location": { "file": "report.docx", "block": "b7" }
    }
  ]
}

Exit codes: 0 when the gate passes, 3 when the gate trips. Use --no-fail to always exit 0 in a fix-loop where you want to inspect findings without aborting the script. Use --lenient for report-only mode.

See Checks for the full rule taxonomy and gate configuration.

Applying edits faithfully​

Once a block is flagged or an AI has drafted a revision, kapi rewrite applies a plain-language instruction to the file text:

kapi rewrite --instruction "tighten each paragraph to two sentences" report.docx

Review the change before writing it:

kapi rewrite --instruction "tighten each paragraph to two sentences" \
  --diff report.docx

Edit the file in place and keep a backup:

kapi rewrite --instruction "fix passive voice" --in-place=.bak report.docx

--provider selects the AI provider when the default is not what you want:

kapi rewrite --instruction "use UK spelling" --provider anthropic -i report.docx

See Rewrite tool for provider configuration and the inline-code preservation contract.

Running the loop via MCP​

If your AI assistant is connected to the kapi MCP server, it can drive the same workflow through structured tool calls rather than shell commands:

  • extract_content — parse a file and return content blocks (the MCP equivalent of kapi inspect).
  • check_file — run the check suite and return a kapi.check/v1 Report.
  • rewrite_file — apply a plain-language instruction and write back faithfully.
  • brand_check — score text against a brand voice profile.

A typical agent loop: the assistant calls extract_content to read blocks, check_file to find findings, then calls rewrite_file on the flagged block, and runs check_file again to confirm the finding is resolved.

See MCP server for setup and the full tool list.

In a project​

In a .kapi project, ingestion runs against the recipe's declared content set — kapi stats, kapi inspect, and kapi check with no file argument survey every source the project declares, instead of paths you list by hand. A project also makes the write-back loop durable: edits applied with kapi rewrite land on the recipe's content, and the same content_hash anchors stay stable across runs, so an AI assistant indexing your project can retrieve and rewrite the same blocks over time. See Create your first project.
