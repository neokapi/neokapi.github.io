Edit content in any format, faithfully

Runs anywhere

No project needed — the commands here run ad-hoc on any source file. No AI provider or credential is required: you supply the edits, kapi applies them.

Goal: edit the text inside a file without touching its structure. The source file may be a Word document, a JSON catalog, a Markdown guide, an HTML page, or any other format kapi reads. The result is the same file — same keys, same styles, same inline codes — with only the prose changed. You do the rewriting; kapi parses the format, enforces a faithful round-trip, and writes the result back. kapi does not send content to a model to rewrite it.

Common uses: simplify technical documentation, fix off-brand copy, standardize terminology across a catalog, tighten prose to fit a length budget.

Assess the file first​

Before editing, it helps to see how much text you are dealing with. kapi stats reads the file and reports block counts, word totals, and character totals without modifying anything:

kapi stats guide.md
kapi stats guide.md --json    # machine-readable output for scripting

Read the blocks​

kapi inspect returns one record per content block — the text, the block's structural role and nesting level, a stable id, and a content_hash:

kapi inspect guide.md --jsonl        # one JSON object per line, for streaming
kapi inspect guide.md -f markdown    # override format detection

Two fields anchor an edit:

  • text renders inline codes (links, bold spans, placeholders) as <x id="…"/> tokens. Keep every token in your edited text — they are the markup the round-trip reconstructs. A placeholder is <x id="1/"/>; a paired span opens with <x id="1"/> and closes with <x id="/1"/>.
  • content_hash is the block's canonical identity (a hash of its plain source text, not of the placeholder text). Send it back with the edit so kapi can tell the block is still the one you read.

kapi inspect does not modify the file.

Edit, then apply​

Rewrite the blocks you want to change, and produce one content entry per changed block — JSONL, one per line — naming the block's file, id, the content_hash you saw, and your new text:

{"kind":"content","file":"guide.md","id":"p3","content_hash":"a1b2c3…","text":"See the <x id=\"1/\"/> dashboard."}

kapi apply is the one write verb. It reads the change-set from a file, an argument, or stdin, and writes each named file in place — no model called:

kapi inspect guide.md --jsonl > blocks.jsonl
# Now rewrite the "text" of each block you want to change — keeping every
# <x id="…"/> placeholder — and save the changed ones as content entries to
# edits.jsonl. (This is the step you, or your AI assistant, do; there is no
# command for it — that is the point: kapi never sends the text to a model.)
kapi apply edits.jsonl --diff           # preview the content changes, write nothing
kapi apply edits.jsonl                    # apply in place
kapi apply edits.jsonl --in-place=.bak    # apply, keeping a .bak

--diff runs the edit and prints a unified diff instead of writing — read every proposed change first. The diff shows only the text that changed; headings, list markers, code fences, and inline codes appear unchanged because kapi does not touch them.

The two guards​

apply writes a block only when both guards pass; a blocked edit leaves that block untouched and is reported, so nothing is silently corrupted:

  • Drift guard. If a block's current content_hash no longer matches the one in your entry, the source changed since you inspected it — the edit is marked stale and skipped.
  • Inline-code guard. If your edited text drops, invents, duplicates, or unbalances an <x id="…"/> token, the edit is rejected rather than written back with broken markup.

Either outcome exits on the gate code (3), distinct from an operational error — a signal to re-inspect the affected blocks and retry with fresh hashes. apply is idempotent: an entry already in the desired state is a no-op, so re-running a partly applied change-set is safe. The same change-set can also carry asset edits (a glossary term, a brand rule), so a content fix and the rule that justifies it land together — see Create content for the mixed change-set.

What "faithfully" means​

Only the extracted prose changes; everything outside it — .docx character styles and table borders, JSON keys and non-string values, Markdown headings and fences — is reproduced byte for byte. This is what makes editing binary office formats (.docx, .pptx) safe; kapi formats list marks such formats faithful in its Edit column. (See Tools for the round-trip guarantee and the format reference for per-format specifics.)

Verify the result​

After editing, run kapi check to confirm the result meets your content rules. The check is deterministic: it reads the file and reports findings without modifying it:

kapi check guide.md
kapi check guide.md --max-chars 500 --forbid "utilize"   # ad-hoc rules
kapi check guide.md --profile company-voice              # a named checkset

The command exits 0 when all checks pass and 3 when the gate fails; the structured report is available via --json. Read the findings, fix the flagged blocks through another apply pass, and re-run until the gate is green. (See Checks for the kapi.check/v1 report model — score, severity-bucketed findings, per-block locations, and the gate.)

In an AI assistant's fix-loop​

When kapi is wired as an MCP server (kapi mcp), an AI assistant drives the same workflow through structured tool calls:

MCP toolWhat it does
extract_contentRead a file's blocks: each block's id, content_hash, source_text with inline codes as <x id="…"/> placeholders, and word_count — the read leg
apply_editsApply a typed change-set (content + asset edits) through the faithful round-trip — the one write verb, no provider
check_fileRun the checkset against a file; returns a kapi.check/v1 Report with per-block finding locations
check_textRun the checkset against a text snippet
brand_checkScore a text against a brand voice profile

A typical agent pass: the assistant calls extract_content (or check_file to read the findings), rewrites the flagged blocks itself, calls apply_edits with a change-set of content entries, then calls check_file again to confirm the gate passes. No model is called for the rewrite — the assistant is the writer, kapi enforces the faithful round-trip and drift-checks each edit by content_hash. See MCP server for setup and the full tool list.

In a project​

In a .kapi project, kapi inspect and kapi check run against the recipe's declared content with no file argument, and kapi apply lands edits against the named files. Folding the check into a flow you run with kapi run makes it repeatable across the whole content set. See Create your first project.
