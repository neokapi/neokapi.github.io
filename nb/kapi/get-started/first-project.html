Your first project

The Quick Start runs kapi ad-hoc — one command, one file, flags for everything. That works for a one-off. For content you touch repeatedly — a product's copy, a docs set, a design system's strings — a .kapi project is the working model: capture the content patterns, quality rules, and flows in a committed recipe once, then run them with short commands and no repeated flags.

A project starts with content quality — measuring, checking, and revising your source. No target languages are required. Localization (producing translated output) is an opt-in step you add when you need it.

1. Initialize the project​

Run kapi init in the directory you want kapi to manage. Supply your source locale; target locales are not required and can be added later:

kapi init --source-locale en
Initialized kapi project "my-app"
  recipe: my-app.kapi
  state:  .kapi

The recipe file (my-app.kapi) is committed to the repository. The .kapi/ state directory is regenerable and belongs in .gitignore.

2. Declare the content​

Tell the project which files to track with kapi add — a glob and an auto-detected format:

kapi add "docs/**/*.md"

kapi ls lists what the project tracks (add --stats for per-file block and word counts). kapi rm removes an entry. For a known stack — for example react-i18next or nextjs — kapi init --framework <stack> pre-fills the content mapping; see kapi presets list --framework.

3. Read the recipe​

The <dir-name>.kapi recipe is a portable YAML file. A content-first recipe declares the source locale and the content to track, without any target languages:

my-app.kapi
version: v1
name: my-app
defaults:
  source_language: en
content:
  - path: "docs/**/*.md"

target_languages is omitted. The project runs checks and rewrites against the source content; there are no locale outputs until you opt in to localization. See the project file reference for the full schema.

4. Measure the content​

kapi stats gives a summary of what the project tracks:

kapi stats --json
{
  "total": {
    "blocks": 312,
    "translatable": 298,
    "non_translatable": 14,
    "words": 18540,
    "characters": 102300,
    "characters_no_space": 86100,
    "segments": 634,
    "by_role": { "body": 201, "heading": 47, "list_item": 50 }
  }
}

kapi inspect extracts one record per content block — useful for feeding an AI assistant, a search index, or an audit script:

kapi inspect docs/guide.md --jsonl
{"file":"docs/guide.md","number":1,"id":"b1a2c3","content_hash":"e3b0c4...","role":"heading","level":1,"text":"Getting started"}
{"file":"docs/guide.md","number":2,"id":"b2d4e5","content_hash":"7f83b1...","role":"body","level":0,"text":"Install kapi from Homebrew or download the binary."}

Pass multiple files or a glob; use -f to override format detection. Without --jsonl, kapi inspect outputs a JSON array.

5. Check content quality​

kapi check runs the configured rule set and reports findings:

kapi check
{
  "schema": "kapi.check/v1",
  "pass": false,
  "summary": { "score": 84, "critical": 0, "major": 3, "minor": 7 },
  "gate": "major",
  "findings": [
    {
      "rule": "terminology.consistency",
      "severity": "major",
      "message": "\"kapi-cli\" should be \"kapi\"",
      "suggestion": "Use the canonical product name throughout.",
      "location": { "file": "docs/guide.md", "block": "b3f5a1" }
    }
  ]
}

The command exits 0 when all findings pass the gate and 3 when the gate is not met. Scope constraints (--max-chars, --max-words, --forbid, --require), named profiles (--profile, --profile-file), and severity thresholds (--strict, --lenient, --no-fail) are all available. A bilingual check — comparing source against a translated file — adds --target FILE --target-lang LANG.

6. Revise the content​

Read a file's blocks with kapi inspect, rewrite the ones you want to change, and write them back with kapi apply — faithfully, preserving format structure, inline markup, and non-text nodes, with no model involved:

kapi inspect docs/guide.md --jsonl > blocks.jsonl
# rewrite each changed block's "text" (keep the <x id="…"/> tags), saving them as
# content entries to edits.jsonl — you (or your assistant) do this, no command
kapi apply edits.jsonl --diff

Review the diff, then apply in place:

kapi apply edits.jsonl

--in-place=.bak keeps a backup alongside each edited file. An edit that drifted or would corrupt an inline code is left unchanged and reported. See Edit content in any format for the content-entry shape and the guards.

Automated check–revise loop via MCP​

When kapi is connected to an AI assistant via its MCP server, the check–revise loop runs without manual steps. The assistant calls check_file to read findings, rewrites the flagged blocks itself, sends them back with apply_edits, then calls check_file again to confirm the gate passes. brand_check runs the full brand voice profile in a single call. check_text operates on inline text rather than a file path. See MCP tools for the complete tool list.

7. Add localization (optional)​

When you want to produce output in other languages, add target locales to the recipe:

my-app.kapi
version: v1
name: my-app
defaults:
  source_language: en
  target_languages: [fr, de]
content:
  - path: "src/locales/en/*.json"
    target: "src/locales/{lang}"
flows:
  translate:
    steps:
      - tool: recycle
      - tool: translate

The target path uses {lang} as a placeholder for the locale code. {name} and {ext} are also available for custom layouts — see the project file reference.

Run and merge​

From anywhere inside the project tree, run a flow by name:

kapi run translate

kapi discovers the nearest recipe by walking up the directory tree (like git), resolves the project's locales and content, and runs the flow. In a project, a run is process-only — results accumulate in .kapi/cache/blocks.db rather than being written to files immediately. Translation memory accrues across runs.

When you're ready to write the localized files, merge replays the stored output to the target paths declared in the recipe:

kapi merge

This writes src/locales/fr/*.json, src/locales/de/*.json, and so on — no flags, because the recipe already says where each locale goes.

Why project mode​

A project trades a one-time init for less repetition and more memory:

  • Defaults captured once. Source locale, content globs, flow definitions, and quality rules live in the recipe. Repeated flags disappear.
  • Quality memory accumulates. The project store keeps a translation memory and termbase that every run reads and writes, so each pass recycles the last one's work.
  • Git-style discovery. Run kapi from any subdirectory and it finds the project automatically — no -p flag, no directory dance.
  • Portable and shareable. The recipe is a committed YAML file; a colleague who clones the repository runs the same flows with the same configuration.

Ad-hoc runs stay available for one-offs, scripting, and CI — the two modes use the same commands with different bindings (where content enters and leaves). See Modes & bindings for the full model.
